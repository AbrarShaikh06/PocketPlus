# **PHASE A-1 — Technical Architecture**

## **1.1 System Overview**

PocketPlus is a mobile-first financial tracking application. Phase 1 is a Flutter Android app backed entirely by Firebase. Phase 2 migrates to a self-hosted Django + PostgreSQL backend after Firebase costs exceed ₹2,000/month or 500+ paying users.

## **1.2 Phase 1 Architecture — Firebase-Native**

**📐 Architecture Rule**

* All Flutter code in Phase 1 communicates exclusively with Firebase SDKs. No custom HTTP server. No REST calls (except Gemini API). Firebase handles auth, database, storage, notifications, and server-side logic via Cloud Functions.

|  |  |  |  |
| --- | --- | --- | --- |
| **Layer** | **Technology** | **Version / SDK** | **Responsibility** |
| Mobile App | Flutter | 3.22+ / Dart 3.4+ | All UI, business logic, local state, offline cache |
| State Management | Riverpod | 2.5+ | Global state, async data, dependency injection |
| Architecture | Clean Architecture + Feature-First | — | Repositories, Use Cases, ViewModels per feature |
| Database | Firebase Firestore | Cloud Firestore v10+ | All structured data: transactions, profiles, connections |
| Auth | Firebase Authentication | v5+ | Google OAuth, Phone OTP, session tokens |
| File Storage | Firebase Storage | v12+ | Receipt photos, temporary PDF blobs |
| Push Notifications | Firebase Cloud Messaging | v15+ | All server-initiated notifications |
| Server Logic | Firebase Cloud Functions | Node.js 20 runtime | Scheduled tasks, Firestore triggers, email sends |
| AI / NLP / OCR | Google Gemini API | gemini-1.5-flash | Voice parsing, receipt OCR, category inference |
| Ads | Google AdMob | google\_mobile\_ads 5+ | Rewarded video, native ads |

# **PHASE A-3 — API Specification**

**📋 API Contract Rules**

* All endpoints are prefixed /api/v1/. All requests require Authorization: Bearer <jwt> except /auth/\* endpoints. All responses are JSON. All error responses use the standard error envelope below. Pagination is cursor-based, not offset-based. All timestamps are ISO 8601 UTC.

## **3.1 Standard Response Envelopes**

**Success Response:**

{

"success": true,

"data": { ... }, // or [] for lists

"meta": { // present on list responses

"cursor": "eyJpZCI6Ii4uIn0=", // opaque base64 cursor

"hasMore": true,

"total": 1250

}

}

**Error Response:**

{

"success": false,

"error": {

"code": "VALIDATION\_ERROR", // machine-readable enum

"message": "amount must be greater than 0", // human-readable

"field": "amount", // present for field errors

"details": { ... } // optional extra context

}

}

**Error Code Enum:**

|  |  |  |
| --- | --- | --- |
| **Code** | **HTTP Status** | **When Used** |
| VALIDATION\_ERROR | 422 | Request body fails schema validation |
| UNAUTHENTICATED | 401 | Missing or expired JWT token |
| UNAUTHORIZED | 403 | JWT valid but insufficient permissions |
| NOT\_FOUND | 404 | Resource does not exist or was soft-deleted |
| CONFLICT | 409 | Duplicate detected (e.g. SMS already logged, invoice number clash) |
| PLAN\_LIMIT\_EXCEEDED | 402 | Free user exceeded profile limit, or Basic exceeded PDF quota |
| RATE\_LIMITED | 429 | Too many requests (see rate limits section) |
| INTERNAL\_ERROR | 500 | Unexpected server error — always reported to Sentry |

## **3.2 Authentication Endpoints**

**POST /api/v1/auth/request-otp**

Sends OTP to phone number. Rate limited: 3 OTPs per phone per 10 minutes.

Request: { phone: '+919876543210' }

Response: { otpToken: 'otp\_abc123', expiresIn: 300 } // 5 min expiry

Errors: VALIDATION\_ERROR (invalid phone format)

RATE\_LIMITED (>3 OTPs in 10 min)

**POST /api/v1/auth/verify-otp**

Request: { otpToken: 'otp\_abc123', otp: '482916' }

Response: {

accessToken: 'eyJ...', // JWT, expires 15 min

refreshToken: 'eyJ...', // JWT, expires 7 days

user: { id, phone, plan, tutorialCompleted, ... }

isNewUser: true // trigger onboarding if true

}

Errors: VALIDATION\_ERROR (wrong OTP), NOT\_FOUND (expired otpToken)

**POST /api/v1/auth/refresh**

Request: { refreshToken: 'eyJ...' }

Response: { accessToken: 'eyJ...' }

Errors: UNAUTHENTICATED (expired or invalid refresh token)

**DELETE /api/v1/auth/logout**

Invalidates the refresh token. Access token expires naturally after 15 min.

Response: { success: true }

## **3.3 Transaction Endpoints**

**GET /api/v1/transactions**

Query params:

profileId : UUID (required)

type : INCOME|EXPENSE (optional)

categoryId : UUID (optional)

dateFrom : Date (optional, YYYY-MM-DD)

dateTo : Date (optional, YYYY-MM-DD)

source : SMS|MANUAL|VOICE|OCR|RECURRING|INVOICE (optional)

cursor : String (optional, for pagination)

limit : Integer DEFAULT 50, MAX 100

search : String (optional, searches merchantName + note)

Response: {

data: [Transaction],

meta: { cursor, hasMore, total }

}

Auth: User can only see their own transactions.

CA can see owner's transactions if ca\_connections.status = ACTIVE.

**POST /api/v1/transactions**

Request body:

{

profileId : UUID (required)

amount : Integer (required, in paise, > 0)

type : INCOME|EXPENSE (required)

source : MANUAL|SMS|VOICE|OCR (required)

categoryId : UUID?

merchantName : String? (max 300 chars)

note : String? (max 500 chars)

transactionDate : Date (required, YYYY-MM-DD)

rawSmsText : String? (only when source=SMS, stored hashed)

smsHash : String? (SHA256 of rawSmsText, checked for dedup)

}

Validation:

- amount must be > 0 and < 10,000,000,00 (1 crore paise = ₹10,00,000)

- transactionDate cannot be in the future (> today + 1 day tolerance)

- profileId must belong to authenticated user

- if smsHash provided: check sms\_dedup\_log — if exists, return CONFLICT

Side effects:

- Writes to audit\_logs

- Writes to analytics\_events: transaction\_created

- Triggers ML categorisation if categoryId is null

- If source=SMS: writes to sms\_dedup\_log

Response: { data: Transaction }

**PATCH /api/v1/transactions/:id**

Allowed fields: amount, categoryId, merchantName, note, transactionDate

NOT allowed: userId, profileId, source, smsHash (immutable)

Side effects:

- Writes to audit\_logs with oldValues and newValues

- Writes to analytics\_events: transaction\_edited

Response: { data: Transaction }

**DELETE /api/v1/transactions/:id**

Soft delete: sets deletedAt = now(), isDeleted = true

Response returns 200 with { data: { id, deletedAt } }

Side effects: writes to audit\_logs with action=DELETE

Undo window: client shows undo toast for 5 seconds.

POST /api/v1/transactions/:id/restore -- reverses soft delete

## **3.5 CA Connect Endpoints**

**POST /api/v1/ca/invite**

Request: { profileId: UUID, caPhone: '+91...' }

Validation:

- profileId must belong to user

- caPhone must be a valid Indian phone number

- no existing ACTIVE or PENDING connection for this profileId

- PRO plan only (returns PLAN\_LIMIT\_EXCEEDED for FREE/BASIC)

Server action:

1. Creates ca\_connections record with status=PENDING

2. Generates inviteToken (64-char random hex)

3. Sends WhatsApp message via wa.me deep link (client-side)

OR sends SMS via Firebase Functions

Response: { data: { connectionId, inviteToken, caPhone } }

**POST /api/v1/ca/accept**

Request: { inviteToken: String }

Validation:

- inviteToken must exist and connection status = PENDING

- inviteToken must not be older than 7 days

- authenticated user's phone must match caPhone on the connection

Server action:

1. Sets ca\_connections: status=ACTIVE, caId=request.user.id, acceptedAt=now()

2. Sends FCM notification to owner: 'Your CA has connected'

3. Writes analytics\_events: ca\_connected

Response: { data: CAConnection }

**DELETE /api/v1/ca/connections/:id**

Revokes CA access. Both owner and CA can call this.

Sets status=REVOKED, revokedAt=now(), revokedBy=OWNER|CA

Sends FCM notification to the other party.

Effect is immediate — Firestore security rules block CA reads instantly.

## **3.6 ML Endpoints (Phase 2 Only)**

**POST /api/v1/ml/categorise**

Request: { merchantName: String, smsText: String?, userId: UUID }

Response: {

data: {

categoryId : UUID,

categoryName: String,

confidence : Float, // 0.0 to 1.0

source : 'user\_model' | 'global\_model' | 'gemini\_fallback'

}

}

Logic:

1. Try user-specific scikit-learn model (if >= 50 corrected transactions)

2. Fall back to global model (trained on all anonymised user data)

3. Fall back to Gemini API if confidence < 0.6

4. Return 'Other' if all fail

Confidence threshold: if confidence < 0.6, never auto-confirm.

Always show to user for manual confirmation regardless.

**GET /api/v1/ml/forecast/:profileId**

Returns next month profit forecast using ARIMA.

Requires minimum 3 months of transaction history.

Response:

{

data: {

predictedProfit : Integer, // paise

confidenceLow : Integer, // paise, lower bound

confidenceHigh : Integer, // paise, upper bound

basedOnMonths : Integer, // how many months of data used

modelType : 'ARIMA' | 'LINEAR' | 'INSUFFICIENT\_DATA'

}

}

**GET /api/v1/ml/health-score/:profileId**

Response: { data: { score: Integer(0-100), breakdown: [...], trend: Float } }

Computed monthly. Cached for 24 hours.

**GET /api/v1/ml/anomalies/:profileId**

Returns flagged transactions for current month.

Response: { data: [{ transactionId, reason, zScore, severity: LOW|MED|HIGH }] }

## **4.1 State Management — Riverpod**

PocketPlus uses Riverpod 2.x exclusively. No Provider, no Bloc, no setState (except inside StatefulWidget for local UI state like animations).

**Provider Naming Convention:**

// Data layer: repositories

return TransactionRepositoryImpl(ref.watch(firestoreProvider));

});

## **4.4 Money Handling — Paise Only**

**💰 Critical Rule**

* NEVER use double or float for monetary values anywhere in the codebase — not in Firestore documents, not in Dart variables, not in API responses, not in Django models. Always use Integer (paise = 1/100 of a rupee). Violating this rule causes silent rounding errors.

// CORRECT

final amount = 45050; // ₹450.50 stored as 45050 paise

final display = amount / 100; // Only divide when displaying to user

// WRONG — never do this

final amount = 450.50; // ❌ floating point

final amount = 450.5; // ❌ floating point

// Currency formatter utility

String formatRupees(int paise) {

final rupees = paise / 100;

return NumberFormat.currency(locale: 'en\_IN', symbol: '₹').format(rupees);

// Output: ₹4,50,000.50

}

## **4.6 SMS Processing Rules**

**📱 Play Store Compliance**

* SMS Read permission requires a Policy Declaration in the Play Console and a privacy policy URL. The exact justification text must be: 'PocketPlus reads bank transaction SMSes to automatically log income and expense entries, eliminating manual data entry for financial tracking.' Review this with Play Store policy before submission.
* SMS is NEVER stored raw in any database. Only smsHash (SHA256) is stored for deduplication
* SMS parsing happens entirely on-device in Dart. No SMS text is transmitted to any server
* Every incoming SMS is checked against sms\_dedup\_log before showing confirmation UI
* If SHA256 hash exists in dedup log: silently skip. No notification. No UI shown
* Malformed SMS (no amount found): silently skip. Log to Firebase Analytics as sms\_parse\_failed
* Unknown bank sender: silently skip. Add to a local 'unknown\_senders' list for future patterns

service cloud.firestore {

match /databases/{database}/documents {

// ── Helper functions ──────────────────────────────────

function isAuthenticated() {

return request.auth != null;

}

function isOwner(userId) {

return request.auth.uid == userId;

}

function hasActiveCAConnection(ownerId) {

return exists(/databases/$(database)/documents/ca\_connections/

$(ownerId + '\_' + request.auth.uid))

&& get(/databases/$(database)/documents/ca\_connections/

$(ownerId + '\_' + request.auth.uid)).data.status == 'ACTIVE';

}

function isValidTransaction(data) {

return data.amount is int && data.amount > 0

&& data.type in ['INCOME', 'EXPENSE']

&& data.source in ['MANUAL','SMS','VOICE','OCR','RECURRING','INVOICE']

&& data.transactionDate is timestamp;

}

// ── Users ─────────────────────────────────────────────

match /users/{userId} {

allow read: if isOwner(userId);

allow write: if isOwner(userId);

}

// ── Profiles ──────────────────────────────────────────

match /profiles/{profileId} {

allow read: if isOwner(resource.data.userId);

allow write: if isOwner(resource.data.userId)

&& request.resource.data.userId == request.auth.uid;

}

// ── Transactions ──────────────────────────────────────

match /transactions/{txnId} {

allow read: if isOwner(resource.data.userId)

|| hasActiveCAConnection(resource.data.userId);

allow create: if isOwner(request.resource.data.userId)

&& isValidTransaction(request.resource.data);

allow update: if isOwner(resource.data.userId)

&& request.resource.data.userId == resource.data.userId

&& request.resource.data.source == resource.data.source;

allow delete: if false; // soft delete only via update

}

// ── CA Connections ────────────────────────────────────

match /ca\_connections/{connId} {

|| isOwner(resource.data.caId);

allow update: if isOwner(resource.data.ownerId)

allow update: if isOwner(resource.data.ownerId) // to mark resolved

**🔴 Production Reality**

## **5.1 SMS Auto-Capture Edge Cases**

|  |  |  |
| --- | --- | --- |
| **Edge Case** | **Detection** | **Handling** |
| Duplicate SMS (same payment, 2 SMSes) | smsHash already in sms\_dedup\_log | Silent skip. No UI. Log as sms\_duplicate\_skipped in analytics |
| Malformed SMS (no amount extracted) | Regex returns null for amount | Silent skip. Log sms\_parse\_failed with sender ID for pattern improvement |
| SMS from unknown bank sender | Sender ID not in known patterns list | Show in 'Unknown SMS' section of auto-capture screen. Let user manually add |
| Amount in words ('five hundred rupees') | Regex doesn't match numeric pattern | Attempt word-to-number conversion (Dart). If fails, show manual entry |
| SMS for credit card bill (not a transaction) | Amount very large, sender is 'CIBIL' or 'AMEX' | Flag as non-transaction. Show dismiss option only |
| Same amount, same merchant, same day (genuine) | smsHash is unique — different SMS text = different hash | Both logged. User can delete one if duplicate |
| SMS received when app is killed | BroadcastReceiver wakes app | FCM-style background handler. Queue confirmation, show notification |
| SMS permission revoked after grant | Permission check fails at parse time | Show one-time 're-enable SMS' prompt in notification. Never crash |
| User has 2 SIM cards (dual SIM phone) | Both SIMs receive SMSes | Route both. Profile mapping handles which SIM → which profile |

|  |  |
| --- | --- |
| **Edge Case** | **Handling** |
| Photo too blurry | Gemini returns null for amount. Show: 'Bill unclear — please enter manually'. Show manual form. |

## **5.3 CA Connection Edge Cases**

|  |  |
| --- | --- |
| **Edge Case** | **Handling** |
| CA accepts invite on different device | inviteToken is device-agnostic. Lookup is by token only. Works on any device. |

# **PHASE B-3 — Security Specification**

## **B3.1 Authentication Security**

|  |  |  |
| --- | --- | --- |
| **Control** | **Implementation** | **Rationale** |
| OTP rate limiting | 3 OTPs per phone per 10 minutes. Firebase Auth built-in + additional Cloud Function check | Prevents OTP bombing / account enumeration |
| JWT access token expiry | 15 minutes. Non-negotiable. | Limits blast radius of token leak |
| JWT refresh token expiry | 7 days. Stored in Flutter SecureStorage (not SharedPreferences) | SecureStorage uses Android Keystore |
| Refresh token rotation | New refresh token issued on every refresh. Old one invalidated. | Prevents refresh token reuse after compromise |
| Device binding (Phase 2) | JWT includes device fingerprint. Server rejects tokens from different devices. | Prevents token exfiltration attack |
| Biometric lock | local\_auth package. Auto-triggers after 5 minutes inactivity. Cannot be disabled without PIN. | Protects financial data on shared devices |
| Session invalidation | DELETE /auth/logout invalidates refresh token server-side. Firebase Auth sign-out revokes Firebase tokens. | Hard logout on lost/stolen device |

## **B3.2 Data Security**

|  |  |
| --- | --- |
| **Control** | **Implementation** |
| Money amounts in paise | All amounts stored as integers. No floating point. Prevents rounding manipulation. |
| SMS never stored | SHA256 hash stored only. Raw SMS never written to Firestore or logs. |
| Receipt photos — access controlled | Firebase Storage rules: only owner UID can read their own files. URL signed with expiry. |
| Audit log — immutable | Firestore rules: audit\_logs can only be written by Cloud Functions (server auth). Users cannot write or delete. |
| CA — read only enforced at DB level | Firestore security rules enforce read-only. CA client code additionally hides write UI. |
| Soft deletes only | No hard deletes on financial data. deletedAt timestamp set. Data retained for 7 years (GST compliance). |
| Profile isolation | Every query includes profileId. No query can cross profile boundaries. |
| Invoice number atomic | Generated server-side using Firestore transaction (atomic counter). Client never generates numbers. |

## **B3.3 Compliance Requirements**

* SMS Data Handling: Play Store SMS permission declaration required. Privacy policy must explicitly state SMS is processed locally and never transmitted
* Financial Data Retention: Transaction records retained minimum 7 years (Indian GST audit requirement). deletedAt is soft-delete only
* DPDP Act (India): Users must be able to export all their data (GET /api/v1/users/data-export) and delete their account with 30-day grace period
* GST Categories: category.gstHead field maps to GST rate slabs. Required for CA GST report generation
* Play Store Billing Policy: All in-app purchases must use Google Play Billing. No external payment links inside the app

# **PHASE B-5 — DevOps & Environment Strategy**

## **B5.1 Environment**

PocketPlus is **single-environment**: one Firebase project and one application
(`in.pocketplus.app`), with **no Flutter flavors**. Local testing uses the
Firebase Emulator Suite (demo project `demo-pocketplus`).

|  |  |  |
| --- | --- | --- |
| **Target** | **Source of truth** | **Notes** |
| App (single) | `in.pocketplus.app` | One Android application id, no `dev`/`staging`/`prod` suffixes |
| Firebase project | `lib/firebase_options.dart` + `android/app/google-services.json` | Regenerate both with `flutterfire configure` to point at the project |
| Local tests | `demo-pocketplus` (emulator) | No real project credentials needed |

**⚠️ Environment Rule**

* Firebase project credentials live only in the checked-in `firebase_options.dart` / `google-services.json` (generated, not hand-edited). The Firebase project is **not** selected by a build argument. Only `GEMINI_API_KEY` is injected via `--dart-define`; manage it via GitHub Actions secrets.

## **B5.2 Secrets Management**

# Build command — single environment, no flavors

flutter build apk --release \

--dart-define=GEMINI\_API\_KEY=${{ secrets.GEMINI\_API\_KEY }}

# In Dart code — access safely

const String geminiKey = String.fromEnvironment('GEMINI\_API\_KEY');

## **B5.3 Crash Monitoring**

* Firebase Crashlytics enabled in production and staging. Disabled in development
* All Failure objects caught in repositories are logged to Crashlytics with non-PII context
* PII (phone numbers, transaction amounts, merchant names) must NEVER appear in Crashlytics logs
* Alert threshold: Crashlytics alert fires if crash-free user rate drops below 99.5%
* Sentry added in Phase 2 for Django backend — all 500 errors trigger Sentry alert
